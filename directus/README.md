# Directus schema snapshots

This directory holds the canonical schema snapshot for the Diamond's
self-hosted Directus instance. The snapshot is a YAML file describing every
collection, field, and relation that Directus tracks.

## Files

- `snapshot.yaml` — the current authoritative schema. Generated by the
  `directus-snapshot` GitHub Action from the live Railway instance.

## Workflow

1. Make schema changes in the Directus admin UI at
   <https://directus-cms-production-7e14.up.railway.app>. (For example: add a
   new collection, expose a new column from a Postgres-only table, polish
   field metadata like display templates or interface choices.)
2. When the changes feel ready, run the `directus-snapshot` workflow from the
   **Actions** tab of the GitHub repo:
   - Click **Run workflow**.
   - Leave defaults, or pick a custom branch name.
   - The Action calls `/schema/snapshot?export=yaml` on the Directus instance,
     commits the result to a branch, and opens a PR.
3. Review the diff in the PR and click **Merge** — that becomes the new
   committed source of truth.

## Why snapshot rather than hand-author?

The Directus schema YAML has many default-valued meta keys per field. Capturing
them via introspection is the only way to guarantee byte-perfect round-tripping
against your specific Directus version. Hand-authored YAML routinely fails
`schema apply` because of subtle missing-default mismatches.

## Restoring from a snapshot

If you ever need to replay the schema (rebuilding a Directus instance from
scratch, restoring after corruption, syncing dev/prod):

```bash
# From a host that has the Directus CLI installed and can reach the instance:
DIRECTUS_URL=... DIRECTUS_TOKEN=... \
  npx directus schema apply ./directus/snapshot.yaml
```

Or via the REST API:

```bash
curl -X POST \
  -H "Authorization: Bearer $DIRECTUS_TOKEN" \
  -H "Content-Type: application/yaml" \
  --data-binary @directus/snapshot.yaml \
  "$DIRECTUS_URL/schema/apply"
```

The instance must be at the same Directus version and database vendor as the
one that produced the snapshot. If you're crossing versions, add `?force=true`
and review the diff first via `POST /schema/diff`.

## Notes on the Postgres view `points_with_zones`

Directus does not natively expose Postgres views via its REST/GraphQL API.
The view exists for ad-hoc SQL queries and front-end clients that read
Postgres directly (via PostgREST or Supabase). It will NOT appear in
`snapshot.yaml` and that's correct. To consume the same data through Directus,
join `coordinates` + `zones` + `sub_zones` in the API request or in client
code.
